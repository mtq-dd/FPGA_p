# FPGA 从仿真到上板 — 踩坑与心得

## 项目概况

- **硬件**：米联客 MLKPAI-Fmin01（安路 EG4S20NG88）
- **工具**：TD 5.6.4 + ModelSim
- **项目**：闹钟（BCD 计时 + UART 串口 + 蜂鸣器 + 按键控制）
- **时间**：2026-05-15 晚，约 4 小时

## 踩坑清单（按出现顺序）

### 坑 1：JTAG 驱动不认

**现象**：设备管理器显示 "USB-JTAG-Cable"，TD 的 Download 窗口显示 "No hardware"。

**原因**：Windows 给 JTAG 下载器装了通用驱动，TD 不认。

**解决**：手动替换为 TD 自带的 `dl-cable` 驱动（`cyusb.inf`），路径在 TD 安装目录 `driver\dl-cable win10-x64\`。

**教训**：设备管理器显示正常 ≠ TD 能认。必须装 TD 配套的 Cypress 驱动。

---

### 坑 2：器件型号选错

**现象**：比特流生成成功，下载时提示 "The bit selected is not match with the device"。

**原因**：建工程时选了 `EG4A20NG88`，板子上是 `EG4S20NG88`。差一个字母，比特流格式完全不同。

**解决**：直接改 `.prj` 文件的 XML 内容，将 `EG4A` 改为 `EG4S`，两个 `.prj` 文件（syn_1 和 phy_1）都要改。

**教训**：FPGA 型号必须严格匹配。`EG4A` ≠ `EG4S`，就像 APK 不能装到 iOS 上。修改 `.prj` 后要重新综合+布局+布线+生成，之前的结果全部作废。

---

### 坑 3：系统时钟不是 50MHz

**现象**：串口输出乱码（`\0x榾`），且每 2 秒才输出一次。LED 闪烁频率不对。

**原因**：代码注释和参数里都写了 `50_000_000`，但板子上 P34 引脚的实际时钟是 **25MHz**。demo 工程用了 PLL 倍频，所以注释里的 50MHz 是 PLL 输出频率，不是晶振频率。

**解决**：将 `CNT_MAX` 从 `50_000_000` 改为 `25_000_000`，UART 的 `CLK_FREQ` 同步改为 `25_000_000`。波特率分频从 434 变为 217（25M / 115200 ≈ 217）。

**教训**：
- 不要假定时钟频率，要看原理图或 PLL 配置
- 串口乱码 + 间隔翻倍 = 时钟频率翻倍错误的经典症状
- 25M 和 50M 之间差一倍，所有定时参数全部要改

---

### 坑 4：上电后寄存器初始态不确定

**现象**：烧录后 LED 状态随机，计数器不走，按键有时不响应。按复位键也无效（按键本身没被软件读到）。

**原因**：FPGA 配置完成后，寄存器初始值不保证为 0。`key_debounce` 模块的同步寄存器（`key_r0`、`key_r1`）上电后随机值产生假边沿，触发错误的按键事件，打乱了状态机。没有复位的话，26 位计数器从随机值开始，可能很久才到目标值。

**解决**：在顶层模块加 POR（上电复位）电路——一个计数器在上电后维持约 1ms 的低电平，自动初始化所有寄存器，不依赖外部按键。

**教训**：
- FPGA ≠ 软件：没有 `main()` 给你初始化全局变量
- 每个模块的复位必须可靠，不能指望按键手动复位
- POR 是 FPGA 设计的标配：计数几千个时钟周期后释放复位

---

### 坑 5：嵌套 case 缺 begin/end

**现象**：TD 解析 Verilog 报错，提示模块是 "black box"。

**原因**：Verilog 中 `case` 项本身不创建作用域。当 `case` 里面再嵌套 `case`，并且外层 case 项下面有多条语句时，必须用 `begin/end` 包裹。否则工具不确定语句属于哪个分支。

**解决**：每个非单行的 `case` 项加 `begin/end`，并补 `default: ;`。

```verilog
// ❌ 错误
case (mode)
    2'b01: case (pos) ... endcase  // 无 begin/end
    2'b10: case (pos) ... endcase  // 后面的 alarm_en <= 1 属于哪？
endcase

// ✅ 正确
case (mode)
    2'b01: begin
        case (pos) ... endcase
    end
    2'b10: begin
        case (pos) ... endcase
        alarm_en <= 1'b1;
    end
    default: ;
endcase
```

### 坑 6：蜂鸣器 DC 驱动声音极小

**现象**：闹钟触发时蜂鸣器几乎听不到。

**原因**：蜂鸣器是压电陶瓷片，DC 电平驱动声音极弱。必须用方波（交流）驱动才能响。

**解决**：闹钟响时输出 `cnt_1s[14]`（约 763Hz 方波）代替 `1'b1`。

**教训**：驱动外设前先看它是 DC 还是 AC 器件。压电蜂鸣器、扬声器类都需要交流信号。

## 有效方法论

### 1. 诊断固件法

当复杂设计不工作时，写一个最简单的诊断固件——每个引脚直接映射到可观察的行为：

- 8 个 LED 各有不同的闪烁/常亮模式
- 按键直接连 LED（按下亮）
- 蜂鸣器持续方波

每次只测一个变量。诊断固件 10 分钟写完，省了盲目排查的 2 小时。

### 2. Android 思维迁移对照

| FPGA 概念 | Android 类比 |
|-----------|-------------|
| 综合（Synthesize） | Java → bytecode |
| 布局布线（Place & Route） | dex → APK 打包 |
| 生成比特流（Bitstream） | APK 签名对齐 |
| JTAG 下载 | `adb install` |
| 引脚约束（ADC） | AndroidManifest.xml 权限声明 |
| 时序约束（SDC） | 主线程 16ms 帧率限制 |
| 上电复位（POR） | `onCreate()` 里的初始化 |
| 按键消抖 | `OnClickListener` 的 debounce |
| ILA 抓波形 | Logcat / Profiler |

### 3. TD 全流程速查

```
建工程 → 选 EG4 / EG4S20NG88
  → Add Source (.v 文件)
  → Add ADC (引脚约束)
  → Add SDC (时钟约束)
  → Set Top Module
  → FPGA Flow: Read Design → Opt RTL → Opt Gate
  → Opt Placement → Opt Routing → Gen Bitstream
  → Download → Add .bit → JTAG → Run
```

## 心得体会

**仿真到硬件之间，隔着四堵墙。**

第一堵是工具链。驱动不认、器件选错、软件版本差异——这些跟 Verilog 水平无关，但它们是上板的必经之路。过不去就没有然后。

第二堵是物理现实。仿真里时钟是理想 50MHz、寄存器从 0 开始、按键没有抖动。板子上晶振是 25MHz、寄存器初始值不确定、蜂鸣器 DC 不响。仿真通过的代码上板就崩，往往不是逻辑错，是物理假设错。

第三堵是调试手段的差异。软件有个 bug，加 print 或者断点就行。FPGA 的 26 位计数器卡住了，你既看不到也打不了 log——除非你设计之初就留了观测口（LED 心跳、UART 输出、ILA 抓波形）。

第四堵是心态。软件改一行代码重编几秒，FPGA 改一行要重新综合布线，几分钟起步。错误信息不像编译器那么友好，经常是一句 "black box" 让你自己猜。容易让人想放弃。但这恰是硬件开发的一部分——慢，但每一步都更接近物理世界。

**最小诊断法是我今晚学到的最重要的方法论。** 不要试图在复杂设计里定位 bug。抽出来，做一个只测一个变量的最小系统。10 分钟写诊断，省 2 小时猜谜。

**POR 是 FPGA 的 onCreate。** 软件里从来不用想"main 函数执行前变量是什么"，FPGA 里这事每一块寄存器都要考虑。上电复位不是可选项，是必选项。

这次从零到上板跑通一个自己的项目，核心收获不是闹钟本身，而是这条链路：**自己写的 Verilog → 自己建的工程 → 自己配的引脚 → 综合通过 → 布局通过 → 比特流生成 → 板子跑起来**。这条链路通了，后面加什么功能都是在这个骨架上改。
